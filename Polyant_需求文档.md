## 1. 项目概述

### 1.1 项目愿景

> **知识不应该属于某个特定机构和人，应该是属于全人类的。**

打造一个分布式、永不宕机的全民知识系统。该系统以AI智能体（Agent）为主要使用者，通过P2P网络实现知识的去中心化存储与共享，确保知识自由流通、永久保存。

### 1.2 核心定位

| 维度 | 说明 |
|------|------|
| **服务对象** | AI智能体（Agent）作为主要使用者，人类用户作为管理者/验证者 |
| **运行形态** | 系统服务（Daemon/Service），随智能体一同运行 |
| **网络模型** | P2P分布式网络，类似BT协议的去中心化架构 |
| **数据形态** | 分布式键值数据库，每个条目包含MD文档 + JSON结构化数据 |
| **知识用途** | 为智能体提供操作技能、领域知识、百科词条等 |

### 1.3 核心特性

- **跨平台**: 支持 Windows、Linux、macOS
- **去中心化**: P2P网络，无单点故障
- **自主注册**: 智能体可自主生成公钥并注册
- **分层权限**: 基础用户（只读）→ 正式用户（读写）
- **评分权重**: 社区驱动的知识质量评估
- **分类体系**: 可扩展的知识分类
- **镜像同步**: BT式数据分发，每个节点都是镜像源
- **永不宕机**: 分布式架构确保系统持续可用

### 1.4 智能体接入方式

系统以 **技能（Skill）** 的形式向智能体暴露功能接口：

- 提供一份 **MD说明书文档**，描述如何调用知识库服务
- 智能体（如 OpenClaw、Claude Code 等）通过 Skill 协议调用平台功能
- 与底层大模型无关，任何支持 Skill 协议的智能体均可接入

## 2. 技术可行性分析

### 2.1 技术栈选型

| 组件 | 推荐方案 | 成熟度 | 可行性 |
|------|----------|--------|--------|
| **P2P网络框架** | go-libp2p | 生产级（IPFS生态） | ★★★★★ |
| **本地KV存储** | Pebble (CockroachDB) | 生产级 | ★★★★☆ |
| **全文搜索引擎** | Bleve | 生产级 | ★★★★☆ |
| **跨平台系统服务** | kardianos/service | 生产级 | ★★★★★ |
| **公钥认证** | crypto/ed25519 (标准库) | 生产级 | ★★★★★ |
| **DHT节点发现** | go-libp2p-kad-dht | 生产级 | ★★★★★ |
| **数据同步协议** | 自定义协议（基于libp2p Stream） | 需开发 | ★★★★☆ |

### 2.2 可行性评估结论

**综合可行性评分: 4.5/5 — 高度可行**

Go生态中P2P和嵌入式数据库领域的库已经非常成熟，特别是 go-libp2p 生态提供了从传输层到发现层的完整解决方案。项目的主要工作量在于：

1. **自定义同步协议的设计与实现**（中等复杂度）
2. **各组件的整合与系统化**（中等复杂度）
3. **评分权重算法与用户层级体系的实现**（低-中等复杂度）

### 2.3 关键技术依赖分析

#### go-libp2p 生态

```
go-libp2p (核心)
├── 传输层: TCP / QUIC / WebSocket
├── 加密层: Noise Protocol
├── 多路复用: yamux / mplex
├── 发现层: mDNS (局域网) + Kademlia DHT (全网)
├── NAT穿透: AutoNAT / Relay
└── 自定义协议: Stream-based Protocol
```

**优势**:
- IPFS/Filecoin 等大型项目生产验证
- 纯Go实现，交叉编译无CGO依赖
- 内置DHT、NAT穿透、加密传输
- 支持自定义协议注册

**风险**:
- 依赖链较深，二进制体积偏大（约30-50MB）
- API学习曲线较陡

#### Pebble 嵌入式KV存储

**优势**:
- CockroachDB团队出品，LevelDB/RocksDB的Go替代品
- 写入延迟低，支持数据压缩
- 纯Go实现，跨平台无障碍
- 支持SST导入/导出，便于数据同步

**风险**:
- API相对底层，需要自行封装
- 不内置TTL过期机制（需自行实现）

#### Bleve 全文搜索

**优势**:
- 纯Go嵌入式全文索引
- 支持多种查询类型（Term/Phrase/Fuzzy/Boolean等）
- 支持中文分词（通过自定义Analyzer）
- 支持结果高亮

**风险**:
- 中文分词需要额外配置（可集成 gojieba 或 sego 分词库）
- 大规模数据性能不如Elasticsearch（但初期规模足够）

## 5. 网络协议设计

### 5.1 协议栈

```
┌─────────────────────────────────────────┐
│  应用协议: Polyant Sync Protocol (AWSP) │
├─────────────────────────────────────────┤
│  传输协议: go-libp2p Stream              │
├─────────────────────────────────────────┤
│  加密层:   Noise Protocol                │
├─────────────────────────────────────────┤
│  多路复用: yamux                         │
├─────────────────────────────────────────┤
│  传输层:   TCP / QUIC                    │
├─────────────────────────────────────────┤
│  发现层:   DHT (Kademlia) + mDNS         │
└─────────────────────────────────────────┘
```

### 5.2 AWSP 协议消息类型

协议ID: `/polyant/sync/1.0.0`

| 消息类型 | 方向 | 说明 |
|----------|------|------|
| `HANDSHAKE` | 双向 | 协议握手，交换节点信息和能力 |
| `HANDSHAKE_ACK` | 双向 | 握手确认 |
| `QUERY` | Local → Seed/Peer | 关键词查询请求 |
| `QUERY_RESULT` | Seed/Peer → Local | 查询结果返回 |
| `SYNC_REQUEST` | 双向 | 数据同步请求 |
| `SYNC_RESPONSE` | 双向 | 同步数据响应 |
| `MIRROR_REQUEST` | Local → Seed | 镜像请求（按分类） |
| `MIRROR_DATA` | Seed → Local | 镜像数据传输 |
| `MIRROR_ACK` | Local → Seed | 镜像确认 |
| `PUSH_ENTRY` | Local → Seed | 推送新/更新条目 |
| `PUSH_ACK` | Seed → Local | 推送确认 |
| `RATING_PUSH` | Local → Seed | 推送评分 |
| `RATING_ACK` | Seed → Local | 评分确认 |
| `HEARTBEAT` | 双向 | 心跳检测 |
| `BITFIELD` | 双向 | 本地数据位图（已有哪些条目） |

### 5.3 消息格式

所有消息使用 Protobuf 序列化：

```protobuf
syntax = "proto3";

package polyant.protocol;

message Handshake {
  string node_id = 1;
  string peer_id = 2;
  NodeType node_type = 3;
  string version = 4;
  repeated string categories = 5;  // 本地拥有的分类
  int64 entry_count = 6;
  bytes signature = 7;             // 节点签名
}

enum NodeType {
  NODE_TYPE_LOCAL = 0;
  NODE_TYPE_SEED = 1;
}

message Query {
  string query_id = 1;
  string keyword = 2;
  repeated string categories = 3;  // 限定分类范围
  int32 limit = 4;
  int32 offset = 5;
  QueryType query_type = 6;        // LOCAL / GLOBAL
}

enum QueryType {
  QUERY_TYPE_LOCAL = 0;
  QUERY_TYPE_GLOBAL = 1;
}

message QueryResult {
  string query_id = 1;
  repeated KnowledgeEntrySnapshot entries = 2;
  int32 total_count = 3;
  bool has_more = 4;
}

message KnowledgeEntrySnapshot {
  string id = 1;
  string title = 2;
  string content = 3;
  repeated bytes json_data = 4;    // JSON数据序列化
  string category = 5;
  repeated string tags = 6;
  int64 version = 7;
  string content_hash = 8;
  double score = 9;
}
```

### 5.4 查询流程

```
Agent ──查询──> 本地节点
                   │
                   ├── 命中且结果充足 ──> 返回本地结果
                   │
                   └── 未命中或结果不足
                         │
                         ├── 连接种子节点
                         │     │
                         │     ├── QUERY (keyword, categories)
                         │     │
                         │     ├── QUERY_RESULT (entries)
                         │     │
                         │     └── 将结果缓存到本地
                         │
                         └── 返回合并结果给Agent
```

### 5.5 数据同步流程

#### 5.5.1 新节点首次同步

```
新节点                          种子节点
  │                                │
  │── HANDSHAKE ──────────────────>│
  │<─ HANDSHAKE_ACK ──────────────│
  │                                │
  │── MIRROR_REQUEST ─────────────>│
  │   (categories: 全部分类)        │
  │                                │
  │<─ MIRROR_DATA (batch 1) ──────│
  │<─ MIRROR_DATA (batch 2) ──────│
  │<─ ... ────────────────────────│
  │                                │
  │── MIRROR_ACK ────────────────>│
  │                                │
  │── BITFIELD (本地数据位图) ────>│
  │                                │
  └── 同步完成，开始提供服务 ───────│
```

#### 5.5.2 增量同步

```
本地节点                          种子节点
  │                                │
  │── SYNC_REQUEST ───────────────>│
  │   (last_sync_timestamp)        │
  │                                │
  │<─ SYNC_RESPONSE ──────────────│
  │   (changed_entries)            │
  │                                │
  │── 更新本地数据 ───────────────>│
  │── 更新全文索引 ───────────────>│
  │                                │
  └── 定时重复 ────────────────────│
```

#### 5.5.3 种子节点间同步

```
种子节点A                        种子节点B
  │                                │
  │── SEED_SYNC_REQUEST ──────────>│
  │   (last_sync_timestamp,        │
  │    entry_version_vector)       │
  │                                │
  │<─ SEED_SYNC_RESPONSE ─────────│
  │   (new_entries,                │
  │    updated_entries,            │
  │    deleted_entries,            │
  │    new_ratings,                │
  │    user_updates)               │
  │                                │
  │── 双向合并 ───────────────────>│
  │                                │
  └── 定时执行(如每5分钟) ─────────│
```

**一致性保证**: 在每个同步周期结束时，所有种子节点的数据应达到最终一致性。使用版本向量（Version Vector）检测冲突，冲突解决策略为"最后写入胜出"（Last Write Wins），以 `updated_at` 时间戳为依据。

```
                    ┌─────────────┐
                    │   Internet   │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ 种子节点A │◄│ 种子节点B │►│ 种子节点C │
        │(公网IP)  │ │(公网IP)  │ │(公网IP)  │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │             │             │
     ┌───────┼───────┐    │    ┌───────┼───────┐
     ▼       ▼       ▼    ▼    ▼       ▼       ▼
   ┌─────┐┌─────┐┌─────┐┌─────┐┌─────┐┌─────┐┌─────┐
   │本地A ││本地B ││本地C ││本地D ││本地E ││本地F ││本地G │
   │(公网) ││(内网) ││(公网) ││(内网) ││(公网) ││(内网) ││(内网) │
   └─────┘└─────┘└─────┘└─────┘└─────┘└─────┘└─────┘
```

**关键规则**:
- 种子节点必须运行在公网服务器上
- 本地节点可以在内网或公网
- 本地节点主动连接种子节点（出站连接），无需被外部发现
- 本地节点之间不直接通信（通过种子节点间接交换数据）
- NAT穿透仅在本地节点需要被其他节点主动连接时使用

## 6. 用户体系设计

### 6.1 注册流程

```
┌──────────────────────────────────────────────────────────┐
│                    注册流程                                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  1. 智能体启动 Polyant 软件                              │
│                    │                                     │
│  2. 自动生成 Ed25519 密钥对                               │
│     - 私钥: 本地安全存储（加密）                            │
│     - 公钥: 作为用户唯一标识                                │
│                    │                                     │
│  3. 向种子节点发送注册请求                                 │
│     - 公钥 + 智能体名称 + 签名                             │
│                    │                                     │
│  4. 种子节点验证签名，创建基础用户                           │
│     - 权限: 只读、查询、镜像共享                            │
│     - 无写权限                                             │
│                    │                                     │
│  5. (可选) 邮箱验证升级                                    │
│     - 用户通过邮箱接收验证链接                               │
│     - 验证后升级为正式用户                                  │
│     - 获得写权限                                           │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 6.2 用户层级体系

| 层级 | 名称 | 条件 | 权限 | 评分权重 |
|------|------|------|------|----------|
| **Lv0** | 基础用户 | 自动注册 | 读取、查询、镜像共享 | 0（不可评分） |
| **Lv1** | 正式用户 | 邮箱验证 | Lv0 + 写入、推送条目 | 1.0x |
| **Lv2** | 活跃贡献者 | 贡献≥10条目 且 评分≥20次 | Lv1 + 提交新分类建议 | 1.2x |
| **Lv3** | 资深贡献者 | 贡献≥50条目 且 评分≥100次 | Lv2 + 维护分类体系 | 1.5x |
| **Lv4** | 专家贡献者 | 贡献≥200条目 且 评分≥500次 | Lv3 + 审核条目、管理用户 | 2.0x |
| **Lv5** | 核心维护者 | 由Lv4用户投票选举 | 全部权限 + 系统管理 | 3.0x |

### 6.3 评分权重算法

**条目综合评分计算**:

```
entry_score = Σ(rating.weighted_score) / Σ(rating.weight)
```

其中:
```
weighted_score = rating.score × rater_weight
```

**rater_weight 由评分者层级决定**:

| 评分者层级 | 权重 |
|-----------|------|
| Lv1 | 1.0 |
| Lv2 | 1.2 |
| Lv3 | 1.5 |
| Lv4 | 2.0 |
| Lv5 | 3.0 |

**排序规则**: 查询结果按 `entry_score` 降序排列，评分越高的条目排序越靠前。

### 6.4 邮箱验证机制

```
智能体主人                    种子节点
    │                           │
    │── 1. 提交邮箱地址 ───────>│
    │   (公钥签名)               │
    │                           │
    │<─ 2. 发送验证邮件 ────────│
    │   (含一次性验证Token)       │
    │                           │
    │── 3. 回传验证Token ──────>│
    │   (通过Agent操作)          │
    │                           │
    │<─ 4. 验证成功，升级权限 ──│
    │                           │
```

**注意**: 验证流程需要人类主人参与确认（通过邮件点击链接或向智能体提供验证码），确保是人类意志的体现。

## 7. 节点类型与行为

### 7.1 节点类型对比

| 特性 | 种子节点 (Seed) | 本地节点 (Local) |
|------|-----------------|------------------|
| **运行环境** | 公网服务器 | 任意设备(公网/内网) |
| **数据范围** | 全量知识库 | 按需镜像(分类级) |
| **网络要求** | 必须有公网IP | 无特殊要求 |
| **服务义务** | 必须提供完整镜像服务 | 可选是否提供镜像 |
| **写入权限** | 接收并审核推送 | 本地自由读写 |
| **数据一致性** | 种子间最终一致 | 与种子节点同步 |
| **部署方式** | 专门部署种子节点软件 | 随智能体自动安装 |
| **数量** | 1台或多台 | 不限 |
| **初始化** | 部署后镜像全量数据 | 自动建立最小知识库 |

### 7.2 种子节点行为

```
种子节点启动流程:
1. 加载本地全量数据库
   - 首次部署时: 导入系统开发方提供的默认数据集
   - 默认数据集随种子节点安装包附带，包含初始分类体系和核心知识条目
   - 后续启动时: 加载已有数据库
2. 启动内置SMTP服务（用于邮箱验证邮件发送）
3. 向已知种子节点列表注册
4. 与其它种子节点执行数据同步
5. 开始监听P2P网络
6. 提供查询、镜像、推送、邮箱验证服务
7. 定时(每5分钟)与其它种子节点同步
```

**种子节点间同步协议**:

```
同步阶段定义:
- SYNC_PHASE_EXCHANGE: 交换版本向量
- SYNC_PHASE_MERGE: 合并差异数据
- SYNC_PHASE_RESOLVE: 冲突解决
- SYNC_PHASE_VERIFY: 数据一致性校验
- SYNC_PHASE_COMPLETE: 同步完成

目标: 在每个同步周期结束时，所有种子节点的数据达到一致状态。
```

### 7.3 本地节点行为

```
本地节点启动流程:
1. 初始化本地最小知识库(内置核心条目)
2. 生成/加载Ed25519密钥对
3. 向种子节点注册(如尚未注册)
4. 连接已知种子节点
5. 根据配置执行选择性镜像同步
6. 开始提供本地查询服务
7. 启动定时增量同步
```

**本地节点的自主决策**:

智能体可以自主决定以下行为（无需人类主人干预）：

| 决策项 | 说明 |
|--------|------|
| 镜像分类 | 根据使用频率和兴趣，选择镜像哪些分类的数据 |
| 镜像开关 | 决定是否将本地数据库共享给其它节点 |
| 本地写入 | 在本地创建知识条目（经过实操验证的知识） |
| 远程查询 | 本地未命中时，查询种子节点并缓存结果 |
| 带宽控制 | 设置镜像服务的带宽限制 |

**需要人类主人确认的行为**:

| 行为 | 说明 |
|------|------|
| 首次注册 | 确认允许在网络上注册 |
| 邮箱验证 | 人类主人提供邮箱并完成验证 |
| 数据推送 | 正式用户推送条目到全量知识库 |
| 镜像共享 | 确认是否允许其他节点从本地下载数据 |

### 7.4 最小知识库

新安装的本地节点自动创建一个最小知识库，包含：

- **系统说明**: Polyant的使用指南
- **核心分类定义**: 初始分类体系
- **示例条目**: 几个示例知识条目，展示数据结构
- **协议说明**: Skill接口的使用说明

最小知识库大小预估: < 1MB

## 9. 非功能需求

### 9.1 性能要求

| 指标 | 目标值 |
|------|--------|

| 远程查询响应时间 | < 2s (正常网络条件) |
| 单节点存储容量 | 支持100万+条目 |
| 并发查询 | 支持100 QPS (本地节点) |
| 种子节点并发 | 支持1000 QPS |

| 指标 | 目标值 |
|------|--------|
| 数据完整性 | SHA-256校验，100%准确 |
| 种子节点可用性 | 多种子冗余，单节点故障不影响服务 |
| 本地数据安全 | 写入WAL（Write-Ahead Log），崩溃不丢数据 |

### 9.3 安全要求

| 指标 | 说明 |
|------|------|

| 权限控制 | 基于用户层级的RBAC |
| 数据防篡改 | 每个条目content_hash + 创建者签名 |
| 防滥用 | 速率限制、条目大小限制 |

### 9.4 跨平台要求

| 平台 | 系统服务形式 | 最低版本 |

### 9.5 资源占用

| 指标 | 目标值 |

## 10. 技术问题与风险

### 10.1 已识别的技术挑战

#### 挑战1: 中文全文搜索的分词质量

**问题**: Bleve默认不支持中文分词，需要集成第三方分词库。

**解决方案**:
- 集成 `gojieba` 或 `sego` 中文分词库作为Bleve的自定义Analyzer
- 支持拼音搜索作为补充
- 预估工作量: 1-2周

**风险等级**: 低

#### 挑战2: 种子节点间数据一致性

**问题**: 多种子节点在并发写入时可能出现数据冲突。

**解决方案**:
- 使用版本向量（Version Vector）追踪数据版本
- 冲突解决策略: Last Write Wins（以updated_at为准）
- 定时一致性校验（每15分钟）
- 预估工作量: 2-3周

**风险等级**: 中

#### 挑战3: 恶意节点与数据污染

**问题**: 恶意用户可能推送垃圾数据或伪造评分。

**解决方案**:
- 所有写入操作需要Ed25519签名验证
- Lv3+用户可审核/标记问题条目
- 评分系统自带抗操纵机制（高层级用户权重更大）
- 速率限制防止批量操作
- 预估工作量: 持续迭代

**风险等级**: 中

#### 挑战4: 邮箱验证的人类参与

**问题**: 智能体无法自主完成邮箱验证，需要人类主人参与。

**解决方案**:
- 提供多种验证方式: 邮件链接 / 验证码
- 智能体通过Skill接口向人类主人请求协助
- 验证Token有效期24小时
- 预估工作量: 1周

**风险等级**: 低

#### 挑战5: 大规模数据同步效率

**问题**: 随着知识库增长，全量同步时间会增加。

**解决方案**:
- 增量同步为主，全量同步仅在初始化时执行
- 数据分片 + 并行传输
- 压缩传输（zstd）
- 按分类选择性镜像
- 预估工作量: 2-3周

**风险等级**: 中

#### 挑战6: 跨平台文件路径与权限

**问题**: 不同操作系统对文件路径、权限、服务管理的处理不同。

**解决方案**:
- 使用 `filepath` 包处理路径（自动适配OS）
- 使用 `kardianos/service` 统一服务管理
- 配置文件使用跨平台格式（YAML/TOML）
- 预估工作量: 1周

**风险等级**: 低
