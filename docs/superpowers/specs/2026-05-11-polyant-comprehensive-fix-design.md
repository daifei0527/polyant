# Polyant 项目全面修复设计方案

## 概述

本设计方案旨在系统性地修复 Polyant 项目中的现有问题，补全功能缺失，提升代码质量和测试覆盖率。

## 问题分析

通过项目探索发现以下关键问题：

1. **功能缺失**：`internal/network/sync/sync.go:449` 中缺少 `ListUpdatedAfter` 接口，阻止增量评级同步
2. **测试覆盖不足**：
   - `category_handler.go` 缺失测试
   - `user_handler.go` 缺失测试
   - `node_handler.go` 缺失测试
   - `helpers.go` 缺失测试
   - `admin/static.go` 零覆盖率
3. **整体覆盖率低**：当前测试覆盖率仅 36%

## 设计目标

1. **功能完整性**：实现所有 TODO 项和缺失功能
2. **测试全面性**：为所有处理器添加测试覆盖
3. **质量提升**：将核心模块覆盖率提升至 70%+

## 实施方案

### 阶段1：ListUpdatedAfter 接口实现

**目标**：完成增量评级同步功能

**接口定义** (`internal/storage/store.go`)：
```go
// 在 EntryStore 接口中添加
ListUpdatedAfter(ctx context.Context, after time.Time) ([]*model.Entry, error)
```

**BadgerStore 实现** (`internal/storage/kv/badger_store.go`)：
- 遍历所有条目，检查 `UpdatedAt` 字段
- 返回在指定时间之后更新的条目
- 考虑性能优化（索引或缓存）

**MemoryStore 实现** (`internal/storage/kv/memory_store.go`)：
- 简单的内存过滤实现
- 用于测试环境

**同步逻辑更新** (`internal/network/sync/sync.go`)：
- 移除 TODO 注释
- 使用新接口获取更新的评级
- 实现增量同步逻辑

### 阶段2：测试补全

**目标**：为缺失测试的处理器添加全面测试覆盖

**测试策略**：
1. **category_handler.go 测试**：
   - 测试分类的 CRUD 操作
   - 测试权限控制（不同用户级别）
   - 测试错误处理（无效输入、不存在的分类）

2. **user_handler.go 测试**：
   - 测试用户注册、登录、信息更新
   - 测试邮箱验证流程
   - 测试用户权限和角色管理

3. **node_handler.go 测试**：
   - 测试节点信息查询
   - 测试节点状态管理
   - 测试网络连接相关功能

4. **helpers.go 测试**：
   - 测试辅助函数的各个分支
   - 测试边界条件和错误情况

5. **admin/static.go 测试**：
   - 测试静态文件服务
   - 测试文件不存在的情况
   - 测试权限控制

**测试工具**：
- 使用 `httptest` 进行 HTTP 测试
- 使用 mock 存储接口
- 表驱动测试模式

### 阶段3：覆盖率提升

**目标**：将整体测试覆盖率从 36% 提升到更健康的水平

**策略**：
1. **覆盖率分析**：
   - 使用 `go test -coverprofile` 生成覆盖率报告
   - 识别覆盖率最低的包和函数
   - 优先处理核心业务逻辑的覆盖

2. **边界条件测试**：
   - 为现有测试补充边界条件
   - 测试错误处理路径
   - 测试并发场景

3. **集成测试增强**：
   - 为 P2P 同步添加更多集成测试
   - 测试端到端的业务流程
   - 验证组件间的交互

4. **持续集成**：
   - 在 CI 中设置覆盖率阈值
   - 自动生成覆盖率报告
   - 防止覆盖率回退

**目标覆盖率**：
- 核心业务逻辑：80%+
- API 处理器：70%+
- 存储层：75%+
- 网络层：60%+

## 实施顺序

采用顺序实施策略，确保依赖关系清晰：

1. **首先**：实现 `ListUpdatedAfter` 接口（功能补全）
2. **然后**：补充缺失的测试文件（测试补全）
3. **最后**：提升整体覆盖率（质量提升）

## 风险与缓解

1. **性能风险**：`ListUpdatedAfter` 实现可能影响性能
   - 缓解：添加索引或缓存机制

2. **测试维护风险**：大量新测试可能难以维护
   - 缓解：使用表驱动测试，保持测试简洁

3. **集成风险**：新功能可能影响现有同步逻辑
   - 缓解：充分的集成测试，逐步 rollout
