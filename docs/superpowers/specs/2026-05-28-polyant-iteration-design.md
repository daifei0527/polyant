## 概述

本设计方案基于对 Polyant 项目的全面审计，确认当前开发状态，并规划下一阶段的迭代方向。项目当前处于 v2.0.1，测试全部通过，整体覆盖率约 70-80%。

## 设计方案

### 策略选择

采用**混合策略**：P0 功能优先完成，同时并行处理独立的测试补全工作。

### 阶段划分

```
阶段1 (P0 + 测试基础)  ─────────────────────────────────────►
  ├── 邮箱验证流程完善
  ├── API 认证中间件完善
  └── seed/user/pactl 测试补全 (并行)

阶段2 (P1 功能)  ─────────────────────────────────────────►
  ├── 用户层级自动升级定时任务
  ├── 条目签名验证
  ├── 种子节点初始数据
  └── 搜索排序算法优化

阶段3 (P2/P3 + 质量提升)  ─────────────────────────────────►
  ├── NAT 穿透增强
  ├── 速率限制中间件
  ├── 整体覆盖率提升到 80%+
  └── Protobuf 序列化 (可选)
```

## 阶段1：P0 功能 + 测试基础

### 1.1 邮箱验证流程完善

**当前状态**: 邮件发送已完成，但验证端点未完整实现

**需要完成的工作**:
- 实现验证码生成和存储（带过期时间）
- 实现 `/api/v1/user/verify-email` 端点
- 添加验证码校验逻辑
- 更新用户状态（邮箱已验证）

**数据流**:
```
用户注册 → 发送验证邮件 → 用户点击链接 → 验证码校验 → 更新用户状态
```

**接口设计**:
```go
// 验证码存储接口
type VerificationStore interface {
    Save(ctx context.Context, code *model.VerificationCode) error
    Get(ctx context.Context, email string) (*model.VerificationCode, error)
    Delete(ctx context.Context, email string) error
}

// 验证端点
POST /api/v1/user/verify-email
Body: {"email": "...", "code": "..."}
Response: {"code": 0, "message": "success"}
```

### 1.2 API 认证中间件完善

**当前状态**: 签名验证逻辑需要完善

**需要完成的工作**:
- 完善签名验证逻辑（处理边界情况）
- 添加时间戳过期检查
- 添加重放攻击防护
- 完善错误处理和日志

**签名验证流程**:
```
请求到达 → 提取签名头 → 构造签名内容 → 验证签名 → 检查时间戳 → 检查重放 → 放行/拒绝
```

### 1.3 测试补全（并行）

**目标**: 将 seed/user/pactl 的测试覆盖率从 0%/0%/39.5% 提升到 70%+

**测试策略**:
- 使用 `httptest` 进行 HTTP 测试
- 使用 mock 存储接口
- 表驱动测试模式
- 覆盖正常流程和错误处理

**需要补全的测试**:
| 模块 | 当前覆盖率 | 目标覆盖率 | 主要测试点 |
|------|-----------|-----------|-----------|
| seed cmd | 0% | 70% | 启动流程、配置加载、初始化 |
| user cmd | 0% | 70% | 启动流程、配置加载、连接种子节点 |
| pactl | 39.5% | 70% | CLI 命令、参数解析、错误处理 |

## 阶段2：P1 功能

### 2.1 用户层级自动升级定时任务

**当前状态**: 缺少定时检查升级任务

**需要完成的工作**:
- 实现定时任务调度器
- 实现升级条件检查逻辑
- 实现 Lv4→Lv5 投票机制（可选，P3）

**升级条件**:
| 当前等级 | 升级条件 | 目标等级 |
|---------|---------|---------|
| Lv0 | 邮箱验证 | Lv1 |
| Lv1 | 创建 10 个条目 + 30 天注册 | Lv2 |
| Lv2 | 创建 50 个条目 + 90 天注册 + 平均评分 > 3.0 | Lv3 |
| Lv3 | 创建 200 个条目 + 180 天注册 + 平均评分 > 3.5 | Lv4 |
| Lv4 | 社区投票（P3） | Lv5 |

### 2.2 条目签名验证

**当前状态**: 创建条目时的签名验证未实现

**需要完成的工作**:
- 实现条目内容签名验证
- 在创建/更新条目时校验签名
- 同步时验证远程条目签名

**签名内容**:
```
SHA256(title + "\n" + content + "\n" + category)
```

### 2.3 种子节点初始数据

**当前状态**: 随安装包附带的默认知识库未实现

**需要完成的工作**:
- 设计初始数据格式
- 实现数据导入逻辑
- 创建默认分类和示例条目

### 2.4 搜索排序算法优化

**当前状态**: 需集成评分权重、时效性因子

**需要完成的工作**:
- 实现综合评分算法
- 集成 TF-IDF 分数、评分权重、时效性因子
- 优化搜索结果排序

**排序公式**:
```
final_score = tfidf_score * 0.4 + rating_score * 0.3 + freshness_score * 0.2 + authority_score * 0.1
```

### 3.1 NAT 穿透增强

**当前状态**: API 请求限流中间件未实现

**需要完成的工作**:
- 实现基于 IP 的速率限制

- 配置可调整的限制参数

**目标**: 整体覆盖率提升到 80%+

**策略**:
- 优先提升低覆盖率模块（crypto 64.3%, protocol 69.9%）

**当前状态**: 使用 JSON 序列化

**需要完成的工作**:
- 定义 Protobuf 消息格式

## 测试策略

### 测试金字塔

```
        ┌─────────────┐
        │  E2E 测试    │  ← 少量，验证完整流程
        ├─────────────┤
        │  集成测试    │  ← 中等，验证模块交互
        ├─────────────┤
        │  单元测试    │  ← 大量，验证函数逻辑
        └─────────────┘
```

### 测试工具

- **单元测试**: `testing` + `testify`
- **HTTP 测试**: `httptest`
- **Mock**: 接口 mock（不使用 mock 框架）
- **覆盖率**: `go test -coverprofile`

### 质量门禁

- 所有测试必须通过
- 新增代码覆盖率 ≥ 80%
- 无 `TODO`/`FIXME` 遗留
- 通过 `go vet` 和 `golangci-lint`

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| P0 功能复杂度超预期 | 阶段1 延期 | 及时识别，必要时拆分任务 |
| 测试补全工作量大 | 并行任务阻塞 | 优先补全核心模块测试 |
| 模块间依赖 | 功能开发顺序受限 | 提前识别依赖，调整开发顺序 |
| 性能问题 | 用户体验下降 | 每个阶段进行性能测试 |
